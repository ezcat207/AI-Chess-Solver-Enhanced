Meet the Cast

Three JavaScript files, each with one clear job

This chess AI is built from just three JavaScript files. No frameworks, no build tools — just vanilla JS you can open directly in a browser.

Think of it like a theater production. Three actors, each playing their role, coordinating to make the show happen.

chess-engine.js — The rules referee. Knows every chess rule, validates moves, detects checkmate.
chess-ai.js — The strategic thinker. Searches moves, evaluates positions, picks the best option.
app.js — The coordinator. Connects the UI to the engine and AI, manages game flow.

What Each File Does

⚖️

chess-engine.js

The rule book. It generates all legal moves, checks if a king is in check, handles special moves like castling and en passant. It has no opinion on which move is better — that's not its job.

🧠

chess-ai.js

The strategist. Gets legal moves from the engine, simulates thousands of future positions, scores each one based on material and position, then returns the highest-scoring move. This is where Minimax and Alpha-Beta pruning live.

🎮

app.js

The stage manager. Updates the visual board when moves happen, handles clicks if a human is playing, tells the AI when it's time to think, displays stats like nodes searched and eval bars. It's the glue between what you see and what's computing underneath.

💡
Why Separation Matters

Each file has one job. This isn't just clean code — it's practical. Want to swap in a different AI algorithm? Just replace chess-ai.js. Want to add a mobile UI? Write a new app.js but keep the engine. Separation of concerns means you can change one piece without breaking the others.

How the System Starts Up

When you load the page, these three components need to introduce themselves and get ready. Here's what happens in app.js:

CODE 
let game = new ChessEngine();
let whiteAI = new ChessAI(4, 'aggressive');
let blackAI = new ChessAI(4, 'positional');

renderBoard();
updateEval();
PLAIN ENGLISH

Create a new chess game with the standard starting position.

Create White's AI brain — it'll search 4 moves deep and play aggressively (prioritizing attacks).

Create Black's AI brain — also depth 4, but plays positionally (prioritizing strong piece placement).

Draw the board on screen with all pieces in their starting squares.

Calculate and display the initial evaluation bar (should be 0.0 since it's an equal position).

That's it. Five lines to boot up a chess computer. The heavy lifting — all the move generation, search trees, and evaluation logic — is hidden inside those class constructors.

Why This Setup Works

This three-file architecture isn't arbitrary. It follows a pattern you'll see everywhere in software: Model-View-Controller (or in modern terms, separation of data, logic, and presentation).

⚖️
chess-engine.js = Model

Holds the game state (where pieces are) and the rules (what moves are legal). Pure logic, no UI concerns.

🧠
chess-ai.js = Controller

Makes decisions based on the model. Takes game state, processes it, returns actions.

🎮
app.js = View

Presents everything to the user. Handles rendering, clicks, animations, stats display.

🎯
When You're Steering AI

If you ask an AI assistant to "add a new feature," knowing this structure helps you be specific: "In app.js, add a button that calls engine.generateLegalMoves() and displays them as highlights on the board." That's way clearer than "make it show possible moves" — you've told the AI exactly which file to touch and which function to call.
